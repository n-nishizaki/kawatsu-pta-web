# 川津小学校PTA Webサイト リプレイス 要件定義書

## 1. 背景・目的

| 項目 | 内容 |
|---|---|
| 現行サイト | http://kawatsupta.byonia.net/ |
| 課題① | レンタルサーバー費用がかかっている |
| 課題② | HTMLを直接編集しており更新が困難（非技術者には不可） |
| 目標① | **0円運用**（サーバー費・ドメイン費ともに無料） |
| 目標② | **非技術者でも気軽に更新できる** |

---

## 2. システム構成

```
[更新担当者]
   Google ドキュメントで記事を書く
   写真はドキュメントに直接貼り付けるだけでよい（Drive保存はGASが自動処理）
        ↓
   ドキュメント内の「公開」ボタン（GASカスタムメニュー）を押す
        ↓
[GAS（Google Apps Script）]
   ・ドキュメントを Markdown に変換
   ・インライン画像 → Google Drive に自動保存 → 公開URLに置換
   ・GitHub API 経由で _posts/ に Markdown ファイルを push
        ↓
[GitHub + GitHub Pages（Jekyll組み込み）]
   push をトリガーに Jekyll が自動ビルド → 静的サイトとして公開
```

### 採用技術

| レイヤー | 技術 | 費用 |
|---|---|---|
| CMS（記事作成） | Google ドキュメント | 無料 |
| 画像置き場 | Google ドライブ（GASが自動保存・公開URL取得） | 無料（15GB） |
| 変換・デプロイスクリプト | Google Apps Script (GAS) | 無料 |
| 静的サイトジェネレータ | Jekyll（GitHub Pages 組み込み、追加設定不要） | 無料 |
| ソースコード管理・ホスティング | GitHub / GitHub Pages | 無料 |
| ドメイン | `ユーザー名.github.io/kawatsu-pta-web/` | **無料** |
| **合計** | | **0円** |

### Markdown + Jekyll を採用する理由

- **CSS・フォントが確実に反映される**: GAS が直接 HTML を生成すると Google Docs のインラインスタイルが混入して CSS が上書きされる。Markdown 経由にすれば Jekyll がクリーンな HTML を生成するため CSS が正確に適用される
- **テンプレートの一元管理**: デザイン変更は `_layouts/` の1ファイルを直すだけで全ページ・全記事に反映
- **安定性**: Markdown はシンプルで変換エラーが起きにくい

## 4. 機能要件

### 4.1 記事投稿・更新・削除フロー

#### 記事 ID の仕組み

- **記事 ID = Google ドキュメントの固有 ID**（URL の `d/` 以降の文字列）
- GAS が `doc.getId()` で取得するため、担当者がどこかに入力する必要はない
- ドキュメントのファイル名を変更しても ID は変わらないため、タイトル変更しても同一記事として扱われる
- ドキュメントをコピーすると ID が変わるため、コピーからの流用は**別の新規記事**として扱われる

#### GitHub ファイル名の形式

```
_posts/{公開日}-{ドキュメントID}.md
例: _posts/2026-03-12-1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms.md
```

公開日とドキュメント ID をファイル名に含めることで、GitHub `_posts/` と Drive 画像フォルダのソート順が一致する。

#### 新規公開

1. 担当者は Google ドキュメントで記事を執筆する（写真はドキュメントに直接貼り付け）
   - ドキュメントのファイル名 = 記事タイトル
   - 本文以外に入力するものはない（ヘッダーテーブル等は不要）
2. カスタムメニュー「PTA公開」→「このドキュメントを公開する」を押す
3. ダイアログで公開日を入力（本日日付が初期値として入力済み）
4. 内容確認画面で「公開する」を押す
5. 画像は `PTA-web-images/{ドキュメントID}/` フォルダに自動保存・公開 URL に置換
6. GitHub の `_posts/{公開日}-{ドキュメントID}.md` として push される
7. 数分以内に GitHub Pages に反映される

#### 変更（上書き公開）

1. 既存のドキュメントを編集する（同じドキュメントを開いて編集するだけ）
2. カスタムメニュー「PTA公開」→「このドキュメントを公開する」を押す
3. ダイアログに現在の公開日が初期値として入力済み（変更することも可）
4. `PTA-web-images/{ドキュメントID}/` フォルダの画像を一度全削除し、新しい画像を保存する
5. GitHub 上の `_posts/` をドキュメント ID で検索し、該当ファイルを特定する
   - 公開日が変わった場合：旧ファイルを削除し、新しいファイル名（新公開日-ID）で作成する
   - 公開日が変わらない場合：同一ファイルを上書きする

#### 削除（正規フロー）

1. 対象ドキュメントを開き、「PTA公開」→「この記事を削除する」を押す
2. 確認ダイアログを表示する（誤削除防止）
3. 以下の順で削除処理を行う：
   1. GitHub 上の `_posts/` をドキュメント ID で検索し、該当ファイルを削除
   2. `PTA-web-images/{ドキュメントID}/` フォルダがあればゴミ箱へ
   3. ドキュメント自体をゴミ箱に移動（完全削除はゴミ箱から手動で行う）

#### 孤立記事のクリーンアップ

正規フローを経ずにドキュメントをゴミ箱に入れた場合、GitHub と Drive に記事・画像が残る。
これを解消するために「孤立記事をクリーンアップ」機能を用意する。

1. 「PTA公開」→「孤立記事をクリーンアップ」を押す
2. GAS が GitHub `_posts/` の全ファイルを取得し、各ドキュメント ID に対応する Doc の状態を確認する
   - Doc が存在しない（完全削除済み）→ 孤立
   - Doc がゴミ箱に入っている → 孤立
3. 孤立記事の一覧を表示し、確認後に GitHub + Drive から一括削除する

> **運用推奨**: 月1回程度、担当者がクリーンアップを実行する。

### 4.1.1 GAS スクリプト管理方針

- GAS はコンテナバインド型（Google ドキュメントに紐づく）を採用する
- 接続設定（GitHub Token・Owner・Repo・Branch）は `config.gs` に一元管理する
- `config.gs` はリポジトリに空テンプレートとして含め、実際の値は各自の GAS エディタで入力する
- 記事ドキュメントをコピーして流用する場合、`config.gs` の値はコピーされるため再設定不要
- コピーしたドキュメントは doc.getId() が変わるため、**別の新規記事**として公開される

### 4.2 会員限定コンテンツ

- Google ドライブの「リンクを知っている人のみ閲覧可能」で共有する
- サイトには直接リンクを掲載する（完全非公開ではなくURLによる実質限定）
- PDFにパスワードをかけることも可（任意）

### 4.3 コンタクト

- Google フォームを作成し、サイトに埋め込み or リンクを掲載する
- 送信内容は Google スプレッドシートに自動集積される

### 4.4 掲示板

- **廃止**（静的サイトのため実装不可。必要であれば後日検討）

## 5. 非機能要件

| 要件 | 内容 |
|---|---|
| 運用コスト | **0円**（全て無料サービスで完結） |
| 更新担当者スキル | Google ドキュメントが使えれば更新可能 |
| モバイル対応 | スマートフォンで閲覧できること（レスポンシブ対応） |
| 表示速度 | 静的サイトのため高速（GitHub Pages CDN） |
| SSL | GitHub Pages が自動で HTTPS 対応 |
| バックアップ | Git 履歴が自動バックアップになる |

## 6. コンテンツ移行方針

| 項目 | 方針 |
|---|---|
| 新規記事 | 新サイトに投稿する |
| 既存記事（2017〜） | 移行しない（旧サイトで引き続き閲覧可能） |
| 旧サイト | しばらく並行運用し、移行が安定したら閉鎖する |
| 旧サイトアーカイブ | 閉鎖前に必要なコンテンツを Google ドライブに保存しておく |

## 9. 今後の進め方（実装ステップ案）

1. **GitHub リポジトリ作成** & GitHub Pages 設定
2. **サイトテンプレート作成**（HTML/CSS。現行サイトのデザインを参考に）
3. **GAS スクリプト作成**（Google ドキュメント → HTML 変換 → GitHub push）
4. **動作テスト**（テスト記事で一通りのフローを確認）
5. **Google フォーム作成**（コンタクト用）
6. **Google ドライブ整備**（会員限定ファイルの共有設定）
7. **公開・旧サイト並行運用開始**
8. **担当者へのマニュアル作成・引き継ぎ**
9. **旧サイト閉鎖**
